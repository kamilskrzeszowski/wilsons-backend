# Wilsons HQ — Changelog

Records changes to the Wilsons HQ app going forward. Bump `APP_VERSION` in `server.js` with each release. (Versions before v20 were built earlier and aren't itemised here.)

## v42 — 21 July 2026
**See who's using HQ and on which version, and push a reload to everyone** — Kamil's follow-up to v41: "can we push a reload onto all open versions, and can we see if users are online?"

- **The Users screen now shows a "Last seen · version" column.** A green/amber/grey dot, how long ago each person's browser last checked in, and **which version they're running**, with anything behind the current version flagged in red. The version is the part that matters: it turns "has everyone picked up the update?" from guesswork into a list you can read.
- **New "Reload everyone" button** (admins only, on the Users screen). Every open copy notices within a minute. **Anyone who isn't part-way through typing reloads on their own; anyone mid-entry gets the "Reload now" bar instead** — so you can clear the whole team onto a new version without risking someone's half-finished stocktake. A pushed reload deliberately overrides an earlier "Later", since an admin asked for it specifically.
- **No extra load:** both features ride on the once-a-minute version check v41 already added. The check is authenticated, so the server already knew who was asking — presence costs one row update per person per minute, and needed no separate heartbeat.
- **Honest limitation, and it's the same one for both features:** only browsers running **v41 or newer** check in at all. Someone still on an older copy shows as "not since the update" even while actively using the app, and won't obey a pushed reload. That's unavoidable — old code can't be told to do new things — and it resolves itself after one round of everyone reopening. It also means the list doubles as your chase list: **anyone showing "not since the update" is exactly who still needs to close and reopen.** Expect everyone to be grey for the first minute after deploying; that's correct.
- **Architecture note:** `last_seen`/`last_version` are two additive columns on `users`, stamped from the existing `/api/version` poll (the browser now sends its own version as `?v=`). The push is a counter in `meta` bumped by a new admin-only `POST /api/force-reload`, returned alongside `version` and `dataRev` on the same poll. A **counter, not a timestamp**, on purpose: browsers compare it against the value they saw at load, so a PC with the wrong clock can't cause either a reload loop or a reload that never arrives.
- Verified on a live server and a real browser: the heartbeat records both fields; a second account reporting `v39` renders as "out of date" in red; the counter bumps, and a non-admin gets 403 with the counter left untouched; a pushed reload **actually reloaded a live browser** (confirmed by the page's JS context being wiped) and the fresh page adopted the new counter as its baseline; **three further checks afterwards did not reload again — no loop**; and with a field focused mid-typing the same push showed the banner instead, leaving the typed text intact. Plus the "HQ survives without Planning" guard re-test.

## v41 — 21 July 2026
**Three bugs Kamil reported, plus live updates across everyone's screens.**

- **Fixed: "(deleted ingredient)" on the Mixing Sheet after a Recipe Costing change.** Renaming the Green Pantry premix — or adding any new ingredient — made the mixing sheet show `(deleted ingredient)` where the name should be. **Nothing was ever deleted and no data was lost**; the database was correct throughout, which Kamil confirmed by refreshing. The cause: a costing sync mints a *new* ingredient row, and the app re-fetched the **recipes** afterwards but never the **ingredient list**, so the browser held a recipe pointing at an ingredient it had never heard of and printed the fallback text. It now fetches both, and the correct name appears within about half a second — no refresh.
- **Fixed: couldn't choose an existing supplier in Recipe Costing.** The per-ingredient supplier dropdown only offered suppliers that already had a price quote *for that exact ingredient*, so an ingredient with no quotes yet gave you no way to pick a supplier you already use — "+ add a supplier…" was the only option. It now lists every supplier known to costing, with the ones quoted for that ingredient (and their prices) first and the rest under "Other suppliers".
- **Fixed: people silently running old versions of the app.** The main app page was the only page in HQ served without a `no-store` cache header, so browsers were free to keep serving a saved copy for hours without ever asking whether it had changed. Every other page (Planning, KPI, Specs, Costing) already set this correctly. That one-line omission is the likeliest reason staff ended up on stale versions.
- **New: the app tells you when it's out of date.** It now re-checks the live version every minute and when you switch back to the tab, rather than once at load. If a newer version is live you get a banner with a **Reload now** button (and a **Later** that's respected for 30 minutes). It reloads silently *only* when you're returning to a tab that was in the background — never while you're on the page, so it can't throw away a half-entered filling sheet or stocktake. The two mismatch directions are now told apart properly: "you're behind, reload" for staff, versus the existing "upload incomplete" warning, which only ever meant a half-finished deploy.
- **New: recipe and ingredient changes appear on everyone's screen.** A `dataRev` counter bumps whenever a recipe, ingredient or supplier changes, and rides along on the same version check — one small request answers both questions. When it moves, each browser quietly re-reads the data and redraws. So a mixing sheet left open on a factory screen stops showing a name someone renamed an hour ago. **It deliberately skips the refresh while a field is focused, a modal is open, or a save is in flight**, and picks it up on the next check instead — redrawing under someone mid-entry would be a worse bug than the one being fixed.
- Verified: dataRev advances on add/rename/delete of ingredients and recipes and does **not** move on reads; the cache header is present on the main page; a change made by a second user is picked up automatically by an already-open browser with no reload; the mid-typing guard confirmed against a focused input, a focused textarea and an open modal, including the full defer-then-catch-up sequence with the user's typed text left intact; both mismatch directions show their correct distinct message; the banner renders in brand colours and pushes the page down rather than covering the nav bar. Plus the "HQ survives without Planning" guard re-test.
- **After deploying this one, ask everyone to close and reopen Wilsons HQ once.** Anyone currently on an old copy is running old code that has none of this — they need one manual refresh for the new mechanism to start protecting them. After that it looks after itself.

## v40 — 19 July 2026
**Fix: email → task import has been silently failing since v35** — a real bug, found by Kamil tagging a real email and it not becoming a task, then pulling the Railway deploy logs at my request.

- **Root cause:** the query Wilsons HQ sent to Microsoft Graph every 5 minutes to find tagged emails — `categories/any(c:startswith(c,'Task'))` — is syntactically reasonable-looking OData but Microsoft Graph does not support `startswith()`/`contains()` nested inside a `categories/any()` lambda for mail messages. Every single tick since this query was introduced in v35 (to catch both plain `Task` and `Task: Name` in one search) failed with "The query filter contains one or more invalid nodes" and returned nothing — logged correctly every time, but nobody was looking at the logs until now.
- **Why this wasn't caught earlier:** every automated test for this feature (rightly) mocks the Microsoft Graph call, since there's no safe way to test against a real mailbox from an automated run — so the tests only ever proved the *matching and task-creation logic* was correct, never that the *actual query sent to Microsoft* was valid. Every live browser check this session used placeholder credentials for the same reason. The real query was never exercised until it hit Kamil's real mailbox.
- **The fix:** stop asking Microsoft to filter by category at all — that was always the risky part. Wilsons HQ now asks for recently-received mail only (last 3 weeks, newest first, up to 50 messages) using a plain date filter, which is a completely standard, safe query, and then checks each message's categories itself, exactly as it already double-checked the results defensively before. Nothing about how you tag an email changes.
- Verified: 13 new assertions confirming the actual query sent no longer contains `categories`, `startswith`, or `any(` anywhere in its filter clause, and a further 4 confirming the full create/route/notify behaviour (plain `Task`, `Task: Name` routing, the notification batching from v36) is completely unaffected by the fix. The "HQ survives without Planning" guard re-test.

## v32 — 19 July 2026
**Planning: monthly schedule view (Phase 1 item 5) + Phase 2 in-app reminders** — built together in one session (see `PLANNING-ROADMAP.md`).

**Schedule view:**
- A new **Schedule tab**: a month grid (prev/next/Today navigation) showing a count badge on any day with something due — one-off tasks and recurring routines together — so it's easy to see at a glance if too much lands on one day. - Occurrences are de-duplicated against already-generated real tasks (so a day never double-counts the same routine), and — **a correctness fix found during browser testing, not by the automated API tests** (which only ever queried future ranges) — occurrences are **never synthesised for dates before today**. A past day either has a real generated task or it doesn't; showing a "this should have happened" marker on a day before the routine even existed was actively misleading, not just redundant. Regression-tested with a mixed past/future range.

**Phase 2 — in-app reminders:**
- The existing "Your planning" card on the Today dashboard (v24) now also shows a **"due soon" (next 3 days)** count alongside overdue/due today, and lists up to 6 upcoming tasks instead of 4.
- **New opt-in browser notification**: a "🔔 Notify me" button on the same card. Turn it on once (asks browser permission, no server involved) and get a native notification when something on your own list is due today — checked at most once per calendar day, using the app's existing local-date helper (never `toISOString()`, which is UTC and would misfire right around a BST midnight — the same class of bug this app has fixed before). No server, no email, no new dependency.
- **A real bug found and fixed during browser testing:** the notification code called a `toast()` helper that doesn't exist in `index.html` (it was only ever defined in `planning.html`) — every click on "Notify me" silently threw `ReferenceError: toast is not defined`. Fixed by switching to `alert()`, matching this file's actual, exclusively-used feedback convention (85 existing call sites) rather than introducing a second, inconsistent feedback pattern.
